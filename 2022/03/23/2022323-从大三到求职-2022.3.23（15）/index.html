今天学习的内容：

  1. Netty

一、Netty

Netty 是一个 基于NIO 模型的高性能PRC框架，我们可以认为他是对NIO模型的一个封装，提供了简单易用的api去快速开发网络程序。

1、三大组件

Channel 和 Buffer

Java NIO 的核心就在于 通道（Channel） 和 缓冲区 （Buffer）

通道就表示 打开到 IO设备 的连接。如果需要使用NIO系统，就需要获取用于 连接IO设备的通道 以及 容纳数据的缓冲区。

然后操作缓冲区对数据进行处理。

简单来说，通道负责传输，缓存负责存储以及后续的处理

常见的Channel有以下四种，其中FileChannel主要用于文件传输，其余三种用于网络通信

  • FileChannel
  • DatagramChannel
  • SocketChannel
  • ServerSocketChannel

Buffer有以下几种，其中使用较多的是ByteBuffer

  • ByteBuffer
    • MappedByteBuffer
    • DirectByteBuffer
    • HeapByteBuffer
  • ShortBuffer
  • IntBuffer
  • LongBuffer
  • FloatBuffer
  • DoubleBuffer
  • CharBuffer

可以看到缓冲其实就是针对不同的基本数据类型的缓冲，ByteBuffer最为常用

Selector

使用多线程技术

原来我们在没有NIO之前，我们每一个新来的请求（一个新的连接），我们都需要去拉起一个新的线程来处理。

如果请求比较少，倒没有什么问题，但是如果请求非常多，服务器端需要启用的线程也会越来越多，消耗也会越来越大。

在实际当中只要并发到1000上下响应请求就会很慢了,所以这种模型是有问题的,这种也就是同步阻塞IO编程（JAVA BIO）

还有如下缺点：

  • 内存占用高
    • 每个线程都需要占用一定的内存，当连接较多时，会开辟大量线程，导致占用大量内存
  • 线程上下文切换成本高
    • 因为有可能出现阻塞，这时候就需要保存当前变量和状态，需要进行上下文切换，这是很消耗资源的
  • 只适合连接数少的场景
    • 连接数过多，会导致创建很多线程，从而出现问题

类似饭店服务员，如果每个客人都配一个服务员且这个服务员要一直服务一位客人直到结束，虽然服务很好，但是开销非常大，顾客多了饭店承担不起

那我们会想到，如果我们用线程池不就可以解决这个问题了吗？

线程池版设计

但是这个模式的缺点也是很明显的，因为 socket工作在阻塞模式下，一个线程就只能处理一个连接上的 socket，比如图中第一个线程，如果在处理socket1时，就不能处理socket3

所以这种模式只适合做可以及时发送请求并且能及时断开的场景，也就是短连接场景

这种方法存在以下几个问题

  • 阻塞模式下，线程仅能处理一个连接
    • 线程池中的线程获取任务（task）中，只有当前执行完任务之后，才会去获取并执行下一个任务（所以如果这个线程进来什么也不做，他也只能干等他结束才可以去执行下一个任务）
    • 如果socket连接一直没断开，则其对应的线程无法处理其他socket连接
  • 仅适合短连接场景
    • 短连接即建立连接发送请求并响应后就立即断开，使得线程池中的线程可以快速处理其他连接
Selector版设计

selector 的作用 就是配合一个线程去管理多个 channel ，获取这些 channel 上发生的事件，这些channel工作在非阻塞模式下，不会让 线程 像之前一样吊死在一个 channel 上，如果一个channel闲置了没有事件，selector会让线程去处理其他的channel。

这种模式适合连接数非常多，但是流量低的场景

调用 selector 的 select（）方法会阻塞知道channel发生了读写就绪事件，这些事件发生，select方法就会返回这些事件交给线程处理。

就是在准备好了才叫 线程去处理，不会浪费时间

ByteBuffer

使用案例

  • 获取一个channel，读取文件并使用getChannel()方法来获取通道
  • 调用 ByteBuffer.allocate()方法来获取Buffer缓冲区，参数为缓冲区的大小
  • 调用buffer 的 flip()方法，来切换为写模式
  • buffer.hasRemaining()方法可以知道当前缓冲里面是否还有元素
  • 调用 channel的 read（）方法，可以把数据从channel中读取数据并写入buffer，参数就是缓冲区buffer，该方法返回值就是表示这次读到的实际字节数，如果返回值为-1，就代表channel中已经没有元素
  • 使用 buffer.get()方法，可以从buffer中读取数据
内部结构和重要参数

字节缓冲区的父类Buffer中有几个核心属性，如下

  • capacity:缓冲区的容量。通过构造函数赋予，一旦设置，无法更改
  • position:下一个读写位置的索引。缓冲区的位置不能为负，且不能大于limit
  • limit：缓冲区的界限。位于limit后面的数据不可读写。缓冲区的限制不能为负，且不能大于容量
  • mark：记录当前position的值。position被改变后，可以通过调用reset() 方法恢复到mark的位置。

以上四个属性必须满足以下要求

mark <= position <= limit <= capacity

在最开始buffer中还没有值的时候，position从0开始，Limit和Capacity在同一个位置（队尾）

在开启了写模式下，比如我们写入四个字符，那么Position就往前移动4位，同时他最大不能超过Limit（就是不能超过容量上限）

这时候我们进行 filp操作，变为读模式，Position重新回到0，Limit变为刚刚插进来的数据的队尾位置（其实很好理解，就是读肯定不能读到没数据的地方）

读取完后的状态

如果又有新数据来了，那么就调用 clear ，回到初始状态,回到写模式

特殊情况：调用 compact方法 压缩数据

比如我们现在只读了a b 两个数据，但是cd还没来得及读，就要切换回写模式，这时候就把未读完的数据向前压缩，让Position指针在未读数据之后，然后再切换到写模式，这样就可以在下次读的时候再把上次没读的数据读完

核心方法
put( )方法

  • put( )方法可以把一个或多个数据数据放入缓冲区中。

  • 进行该操作后，position的值会根据传入数据的数量增加相应的值，并且指向下一个可以放入的位置，比如我们插入3个值，那么position的位置就在第四个空位上。capacity = limit ，为缓冲区容量的值。

get( )方法
  • 这个方法可以从buffer中取到一个值，根据position位置来取值
  • 进行该操作后，position会+1，如果超过了limit则会抛出异常
  • 注意：get(i)方法不会改变position的值

但是如果我们刚put完就直接get，这明显是不行的，因为这时候的position指在空白的位置上，get只能获得0值

所以我们需要先调用 flip( )方法，去改变模式为读模式，从而重置 Position的位置并且把Limit的位置置为数据之后

flip()方法

从前面我们可以知道，flip方法的主要作用就是切换缓冲区的读写模式

  • flip()方法会切换对缓冲区的操作模式，由写->读 / 读->写
  • 进行该操作后
    • 如果是写模式->读模式，position = 0 ， limit 指向最后一个元素的下一个位置，capacity不变
    • 如果是读->写，则恢复为put()方法中的值
rewind()方法
  • 该方法只能在读模式下使用
  • rewind()方法后，会恢复position、limit和capacity的值，变为进行get()前的值

如果我们需要重复读取，就使用这个方法

clean()方法
  • clean()方法会将缓冲区中的各个属性恢复为最初的状态，position = 0, capacity = limit
  • 此时缓冲区的数据依然存在，处于“被遗忘”状态，下次进行写操作时会覆盖这些数据

compact()方法

此方法为ByteBuffer的方法，而不是Buffer的方法

  • compact会把未读完的数据向前压缩，然后切换到写模式
  • 数据前移后，原位置的值并未清零，写时会覆盖之前的值

clear() VS compact()

clear只是对position、limit、mark进行重置，而compact在对position进行设置，以及limit、mark进行重置的同时，还涉及到数据在内存中拷贝（会调用arraycopy）。所以compact比clear更耗性能。但compact能保存你未读取的数据，将新数据追加到为读取的数据之后；而clear则不行，若你调用了clear，则未读取的数据就无法再读取到了

所以需要根据情况来判断使用哪种方法进行模式切换

mark()和reset()方法
  • mark（）是把当前position的值保存到mark属性中
  • reset（）方法是会把position的值修改为mark中的值

图中我们可以看到在get了前两个元素后，执行了mark()方法，后面调用reset()方法时，就从mark记录的第二个元素开始读取

get(i)方法

他和普通get的区别在于不会改变position的位置，而是直接返回参数下标处的数据

两个buffer初始化的区别

第一个读写效率低，第二个分配效率低，且有可能造成内存泄露。
