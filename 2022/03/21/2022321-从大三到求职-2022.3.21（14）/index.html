Happens-Before原则

此原则用于检测某操作是否能进行重排序，而重排序是编译器对JVM字节码进行重排序的操作在这里关于“操作”两个字定义的很模糊（相当于没有定义），那什么是操作呢，什么又是一个不可再分的原子操作呢；对JVM层面来说，一个JVM字节码就是一个对于JVM执行指令不能再分的原子操作，且上下字节码的操作结果完全互相可见，再往下可能会有更细粒度的字节码以及机器指令，但是现在只讨论JVM字节码；在java语言中，对基本变量int char等的读写是一个原子操作，编译成字节码只有一条(这才是原因)，因此很多书上经常写关于基本变量的读写操作来作为例子，很久之前我还以为一条java代码是一个原子操作。。。

在这里定义的操作的含义就是一条java代码（有时候还需要根据具体语境具体分析其到底指的是什么）

仅仅一个new Object（）对象都有好几条JVM字节码与之对应，更别说一个复杂耗时的大对象的new或者初始化了，上面讲到，当两个操作不满足Happens-before时，JVM将对其进行重排序，对于最上面的那个DCL例子而言，线程A最先进来发现resource为null,因此进入同步块，再次观测resource==null,此时进行初始化，接下来的操作JVM可能会对其进行重排序，

3、Docker Compose

Docker Compse 的作用是对Docker容器集群的快速编排。

比如我们一个项目，除了项目本身，还需要 类似 mysql ，redis，es，mq等中间件的容器，这时候Docker Compose 就可以派上用场。

这样如果正常我们想启动app这个镜像，我们还需要把每个中间件单独启动，这是非常麻烦的

如果我们把项目镜像和这些中间件打包在一起，每次启动全部一起跟着启动，这样不就更好吗

Compose正是为了解决这个问题

Compose 的定位是 定义和运行多个Docker容器的应用 ，它允许用户通过一个单独的 docker-compose.yml 模板文件来定义一组相关联的应用容器为一个项目。

Compose 中有两个重要的概念：

  • 服务 (service)：一个应用的容器，实际上可以包括若干运行相同镜像的容器实例。（比如我们使用docker file 构建的容器 ）
  • 项目 (project)：由一组关联的应用容器组成的一个完整业务单元，在 docker-compose.yml 文件中定义。（就是我们画好框的部分）

所以以后我们项目中每一个项目都应该有独立的 docker-compose.yml

只要有这个文件，docker compose 就认为你是一个项目

Compose 的默认管理对象是项目，通过子命令对项目中的一组容器进行便捷地生命周期管理。

Compose 项目由 Python 编写，实现上调用了 Docker 服务提供的 API 来对容器进行管理。因此，只要所操作的平台支持 Docker API，就可以在其上利用 Compose 来进行编排管理。

5、HashMap

我之前一直知道HashMap的底层是 Node数组 + 链表或红黑树，但不清楚他们分别用来存什么

HashMap的主干是一个 Entry 数组 ，Entry 是HashMap 的基本组成单元，每一个Entry 数组都包含一个K-V键值对

数组是HashMap的主体，链表则是主要为了解决哈希冲突而存在的。

当我们在往里 put元素 的时候，添加的元素虽然key值可能不同，但是他们的hash码可能是相同的，这就是哈希冲突

所以在添加元素时，首先判断是否存在hashcode相同的key元素，如果存在相同的key和相同的hash，则新值覆盖老值，返回旧值。

如果key不同，hashcode相同，就产生hash冲突，使用链地址法解决

链表如何解决hash冲突？

key：put方法的key。

value：put方法的value。

next：因为存在相同的hashcode，不同的key，所以next是下一个entry的地址，也就是位置。

每一个entry数组都有可能产生链表，只要产生了哈希冲突，都会在后面形成一个链表来储存发生冲突的值。

为了解决冲突，一个hash中存放的链表，元素的实际位置在数组的链表上，所以相同hashCode的不同元素可以存储在同一位置。

也就是说，在发生冲突的 entry数组中，以相同的hashcode的key 为 key，然后真正的value 存在链表中。

有的时候冲突可能比较多，所以此时的查询一个 key 的过程会变成了，先计算 key 的哈希值，然后发现这个值对应的位置是一条链表，然后再遍历这个链表，直到找到（或者找不到）对应的 key。

这样一来，如果冲突很多的话，多了这个遍历链表的过程，查询效率就降低了，所以在 JDK1.8 之后引入了一个优化，如果这个冲突的链表比较多的话（貌似是多于 8 个），就把这个链表转换成红黑树，这样的话，可以在 M 个冲突的值中使用 O(logM) 的时间复杂度找到对应的 key。
